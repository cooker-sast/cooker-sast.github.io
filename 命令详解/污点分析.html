基于k-call context sensitive的指针分析的污点分析

cooker的污点分析，在处理方法调用的时候，是这样的处理思路：能指针分析就指针分析，不能指针分析就cha，还不行就人工适当干预，总之call graph必须建起来，方法调用链路不能断。这样做可能会使call graph中某些部分是不精确的，但是必须接受这种不精确带来的误报和冗余的分析，以求能够覆盖真正的漏洞触发点。

# 运行（Run）

java -jar cooker.jar taint --acp java-sec-code-1.0.0.jar --taint-config $HOME/.cooker/taint-config/default.yml --work-dir ta --entry-method-yaml-input controller_method.yml --level 7 --time 60 --complete-dir-output complete --simple-dir-output simple --sink-txt-output sink.txt --taint-flow-dir-output flow
1

controller_method.yml 的内容如下，表示全程序分析的入口函数

entryMethods:  
  - entryClass: "org.joychou.controller.Rce"  
    methodSig: "java.lang.String CommandExec(java.lang.String)"
1
2
3

$HOME/.cooker/taint-config/default.yml 是通用的用来扫扫描 tomcat web应用的taint config

--time 60 表示每一个以controller作为入口的全程序分析，限时运行60s

--level 7 表示用 7-call 的 context sensitive的指针分析，且如果方法调用路径的长度超过7，就不会继续处理该方法调用

# 结果（Result）

file 说明 case
simple 方法调用路径。simple下每一个目录对应的是一个controller的分析结果，每一个文件的文件名代表最终调用危险方法，文件内容是方法调用栈
complete 和simple的结构基本一致，文件内容除了方法调用栈的方法之外，还记录了调用的具体callsite。可以理解为记录了调用时的具体语句，但是这个调用语句的形式是IR，如果想查看这个调用语句在整个类中的具体位置可以把对应类的IR给dump下来研究，方法参考导出ir
flow taint flow，展示污点的流动情况，是一个 list of 变量，表示污点数据是从最上面的变量流到最下面的变量的。污点数据是使用者可控的数据。
sink.txt 分析结束之后整理所有controller都触发了哪些sink

一般情况下，我们可以通过flow的内容查看是哪个来自用户的输入最终跑到了危险方法的参数当中

通过complete或者simple的内容查看方法的调用栈。

complete和flow涉及的callsite和csvar的格式是tai-e ir，如果想要进一步理解结果和调试，可以dump出对应class的ir进行查看，具体方法参考导出ir

# 1. extend

如果只在一个文件里面维护污点分析的规则实在是太不方便了，因而cooker的污点配置文件支持文件之间的引用嵌套，可以通过extend参数包含其他配置文件，extend下面的值可以是绝对路径或者相对路径，相对路径优先会考虑cooker运行目录，然后会考虑resource dir，resource dir的默认位置是 $HOME/.cooker

# 2. sources

sources当中的方法如果被调用，那么污点数据就会流入调用语句的lvalue。比如 id = request.getParameter("id") ，那么污点数据就会流入 id 这个变量。

key value
method 一个method的signature
type 污点对象的类型

# 3. sinks

只有在危险方法被调用，且其中关键参数可控的情况下，才被认为有漏洞。 因而准确来说，一个sink表示的是一个变量。要么是invoke method时的object实例，要么是invoke method时传入的argument。 比如说调用了<java.lang.Runtime: java.lang.Process exec(java.lang.String)> ，还必须要求这个方法的String变量是攻击者可控的，否则即使调用到了也没风险。

key value
method 危险方法的signature
index 可以是base、数字。base表示污点流入调用的实例，数字表示污点流入方法的第n个参数

# 4. transfers

污点传播函数

污点数据除了通过赋值等语句传播之外，可能也会通过一些method进行传播，比如字符串的拼接等等。

key value
method 污点传播函数
from 从哪个变量，可以是base 或者arg数字
to 到哪个变量，可以是result、base、arg数字
type 传播之后生成的新污点数据的类型

# 5. reflect-transfers

cooker的污点分析，在处理方法调用的时候，是这样的处理思路：能指针分析就指针分析，不能指针分析就cha，还不行就人工适当干预，总之call graph必须建起来，方法调用链路不能断。

基于上述的想法设计了reflect object。考虑到有一些method被调用之后，你知道它一定会return一个确定类型的object，但是因为这个method的内部可能不会直接用new语句来生成一个object，而是通过一些其他更为复杂的方式，这就会导致分析的时候到这个method就断了。所以为了解决这个分析过程中的痛点，cooker设计了一个reflect-transfers，其中的方法只要被调用，就会给result一个reflect object，确保继续。

举个例子，比如说针对下面这段代码：

ScriptEngine engine = new ScriptEngineManager().getEngineByName("js");
enine.eval(request.getParmeter("code"));
1
2

如果不用reflect transfers，engine变量不会有任何object流入，想要分析 js engine 的代码注入就进行不下去，增添一个 reflect transfers之后：

reflect-transfers:  
  # ScriptEngine engine = new ScriptEngineManager().getEngineByName("js"); 真没法分析得到 engine,engine 的值是空的，没法继续了，所以用reflect obj，为了能够继续分析  
  - { method: "<javax.script.ScriptEngineManager: javax.script.ScriptEngine getEngineByName(java.lang.String)>", from: base, to: result, type: "javax.script.ScriptEngine" }
1
2
3

这样就保证当调用getEngineByName的时候，一定会有一个javax.script.ScriptEngine对象流入engine变量，这样分析就能顺利进行

同理，对于 <java.lang.Class: java.lang.Class forName(java.lang.String)> 、<java.lang.Class: java.lang.reflect.Method getMethod(java.lang.String,java.lang.Class[])> 等等反射相关的方法也是类似的处理逻辑。

key value
method method signature
from 一般是base
result 一般是result
type 想要返回的object的类型

# --entry-method-yaml-input

name 说明 required default
--entry-method-yaml-input 全程序分析的入口方法，一般是controllers true

# --time

name 说明 required default
--time 每一个全程序分析的最大用时 false 60

如果是-1表示没有时间限制。大部分情况不加--time感觉停不下来。对于某些acp比较大的程序，可以适当增加 --time 大小，对于800M左右的程序，150s可以分析到至少10层。

# --call

name 说明 required default
--call 用了k-call的指针分析 false 7

k-call的指针分析的context为调用栈，--call 表示调用栈的长度 如果不想用 context sensitive的指针分析，可以 --call ci 用context insensitive的分析

# --level-limit

name 说明 required default
--level-limit 是否限制调用栈长度 false true

如果用了k-call的指针分析，--level-limit又为true，则会在分析的时候限制从entry method出发的调用路径的长度。也是确保分析能在可控范围内停止。
